# Recipe: adding a new dataset to `pad_dataset_registry`

Reusable procedure for publishing a new PAD dataset to **[PaperAnalyticalDeviceND/pad_dataset_registry](https://github.com/PaperAnalyticalDeviceND/pad_dataset_registry)** as a Jekyll-rendered catalog entry at `https://padproject.info/pad_dataset_registry/datasets/<DATASET_NAME>/`. Distilled from the two submissions filed from this project on 2026-06-07 (PRs [#4](https://github.com/PaperAnalyticalDeviceND/pad_dataset_registry/pull/4) and [#5](https://github.com/PaperAnalyticalDeviceND/pad_dataset_registry/pull/5) for `Leiberman-Lab_ChemoPADPLStraining2026_Annotated_v1.0`).

The procedure is meant to be executable by a Claude Code session or by a human; each step calls out what the tool will help with and what only the human can do.

> **⚠️ This recipe is the annotated_chemopad project's worked example.** Filesystem references and links throughout (e.g. `scripts/build_registry_release.py`, `data/registry_release/<DATASET_NAME>/`, the wiki) point at this repo, [`PaperAnalyticalDeviceND/annotated_chemopad`](https://github.com/PaperAnalyticalDeviceND/annotated_chemopad), and will not resolve from another project's checkout. If you are submitting from a different source repo, treat this page as a worked example to learn from, and use the **canonical procedural reference** at [How to add a dataset](https://padproject.info/pad_dataset_registry/how-to-add-a-dataset.html) on the registry catalog site for the conventions that apply to all submissions.

---

## Quick mental model

The registry is a GitHub repo with one directory per dataset under `datasets/`. CI on every push to `main` runs `validate_croissant.py` + `generate_catalog.py` and deploys the result to the `gh-pages` branch (note: hyphen, not underscore). GitHub Pages serves `gh-pages` → padproject.info. **There is no PR-time CI** — broken `croissant.jsonld` files surface only post-merge, so always run the validator locally before pushing.

Workflow trigger paths: `datasets/**`, `docs/**`, `**.md`. Anything you change under those triggers a rebuild on merge.

## What you must ship per dataset

Under `datasets/<DATASET_NAME>/`:

| File | Required | Notes |
|---|---|---|
| `metadata_dev.csv` | yes | Training split, standard 8-column schema (+ optional extensions) |
| `metadata_test.csv` | yes | Test split, same schema |
| `metadata_val.csv` | optional but expected | Validation split. The registry accepts it; ship one if you have one |
| `labels.csv` | optional | Single line of comma-separated class names (alphabetical, matches the prior Lieberman entry convention) |
| `projects.csv` | optional | Single line: parent project lineage (e.g. `ChemoPADPLStraining2026`) |
| `class_distribution.csv` | optional | Per-class counts split by `#dev` / `#val` / `#test` / `#total` |
| `dataset_sizes.md` | optional | Two-row table: images + unique sample IDs per split |
| `README.md` | optional, **expected** | Rendered as the catalog page on padproject.info. Markdown only; figs/ images referenced as `./figs/<name>.png` |
| `figs/` | optional, **recommended** | PNG visualizations referenced from the README. `generate_catalog.py` explicitly copies the `figs/` directory into the rendered output path (lines 174–180 of that script as of 2026-06-07) |
| `croissant.jsonld` | yes | Auto-generated by the upstream tool (interactive); commit the result |

### Standard 8-column metadata schema

```
id, sample_id, sample_name, quantity, camera_type_1, url, hashlib_md5, image_name
```

- `id`: int, the upstream PAD image ID (joins to the upstream image server).
- `sample_id`: int, the physical-card identifier (`PAD#`).
- `sample_name`: lowercase hyphen-separated drug class. Match the prior Lieberman entry's convention exactly: `mtx-injectable`, `mtx-oral`, `oxaliplatin-2-mg-ml`, `oxaliplatin-5-mg-ml`, `hydroxyurea` (no `-oral` suffix), `blank`, `lactose`.
- `quantity`: numeric. Integer if all values are whole percentages; otherwise float with one decimal place. The croissant schema declares `xsd:integer` by default; if you use floats, the validator's basic check passes but consumers expecting integers may complain.
- `camera_type_1`: the **raw upstream camera string** (e.g. `iPad`, `HMD Global Nokia 2.3`, `Google Pixel 3a`). Not your analyst-merged bucket — preserve the upstream value here.
- `url`: full HTTPS URL to the image on `pad.crc.nd.edu`. Match the prior entry's double-slash form (`https://pad.crc.nd.edu//var/www/html/...`) if you want exact parity; the single-slash form also resolves.
- `hashlib_md5`: lowercase hex md5 of the PNG file bytes. **Cross-check against the prior Lieberman entry's hash for any shared image** — they should match byte-for-byte, which validates your pipeline.
- `image_name`: `<id>__<sample_id>__<sample_name>__<quantity>.png` with double-underscore separators. Quantity formatted as integer when whole, one-decimal float otherwise.

### Extending the schema

You can append extra columns to the right of the standard 8 (we shipped `camera_bucket`, `lighting`, `background`, `status`, `missing_card`). Strict-schema consumers ignore unknown columns; tools that know them benefit. Document the extensions in the README so consumers know what they mean.

## Procedure

### 1. Build the staging directory locally

Don't write into a clone of `pad_dataset_registry` yet. Stage everything under your dataset's source repo first (e.g. `data/registry_release/<DATASET_NAME>/`) and commit it there for reproducibility. The `pad_dataset_registry` clone is just a place to land the files for the PR.

Use a build script analogous to **[`scripts/build_registry_release.py`](../scripts/build_registry_release.py)** in this repo:

- Reads your source manifest (`PAD#` → metadata) and source CSV (for raw camera string and any annotation columns the manifest lacks).
- Joins on `matched_id_int = matched_id` (be careful — the source CSV may have non-numeric sentinels like `"no_match"`; coerce with `pd.to_numeric(..., errors="coerce")`).
- Computes md5 over every cached PNG.
- Transforms columns per the standard schema mapping.
- Splits into per-split CSVs by your `split` column.
- Writes `labels.csv` (alphabetical), `projects.csv`, `class_distribution.csv`, `dataset_sizes.md`, and `README.md` (templated).

The script in this repo is parameterized only by `--out-dir`; for a different source dataset you adapt the input paths and the column mapping at the top of the file.

### 2. Generate analysis figs (optional but recommended)

Use a script analogous to **[`scripts/build_registry_plots.py`](../scripts/build_registry_plots.py)**:

- Class distribution by split (stacked horizontal bar — readable for 5–20 classes).
- Categorical heatmaps for any annotation axes you ship (camera × drug, lighting × drug, background × drug). Mirrors the prior entry's `camera_drug_heatmap.png`.

Save under `<staging-dir>/figs/<name>.png`. The README references them with `![Alt](./figs/<name>.png)`.

### 3. Pick the dataset name

Follow the convention: `<Lab>_<ProjectFamily>_<Subset_or_Variant>_v<version>`. Examples in the registry:

- `Leiberman-Lab_ChemoPADNNtraining2024_Partial-Drug-Set_v1.0`
- `Leiberman-Lab_ChemoPADPLStraining2026_Annotated_v1.0`
- `FHI360_FHI2020-FHI2022_MidTrainingSet_Good_v1.0`

Notes from the registry:
- "Leiberman" is spelled with `ei` in the registry, even though the proper spelling is "Lieberman". Match the existing convention.
- Year in the project label is the year that's most meaningful (training year for the prior entry; annotation + HPLC re-labelling year for ours).
- Use hyphens within tokens (`Partial-Drug-Set`); underscores separate the major segments.

### 4. Copy the staged directory into the registry clone

If you don't have a clone:

```bash
git clone https://github.com/PaperAnalyticalDeviceND/pad_dataset_registry.git ../pad_dataset_registry
```

Then:

```bash
cd ../pad_dataset_registry
git checkout main && git pull
git checkout -b <DATASET_NAME>-add   # or similar
cp -r /path/to/your/staging/<DATASET_NAME> datasets/
```

### 5. Generate `croissant.jsonld`

The upstream generator is `docs/_scripts/create_croissant.py` and asks three interactive questions: description, version, creator name. Pipe answers via stdin to keep it scriptable:

```bash
printf 'YOUR DESCRIPTION HERE\nv1.0\nFirstAuthor, SecondAuthor, ThirdAuthor (Lab, Affiliation)\n' \
  | uv run --no-project --with pandas --with jsonschema --with requests \
    python docs/_scripts/create_croissant.py \
      --dataset-dir datasets/<DATASET_NAME>
```

The `--no-project` flag is **critical**: without it, uv tries to install the registry's full `pyproject.toml` (which includes GPU-only deps that won't install on macOS) and the script fails.

Output: `datasets/<DATASET_NAME>/croissant.jsonld`. The generator embeds the dataset name in the `@id` URL and in `distribution[].contentUrl` paths, so you must regenerate if you ever rename the dataset directory.

### 6. Validate locally

```bash
uv run --no-project --with pandas --with jsonschema --with requests \
  python docs/_scripts/validate_croissant.py
```

The script loops through every entry under `datasets/`. Your new entry should print `✅ <DATASET_NAME>: Basic validation passed`. If the upstream Croissant schema URL is unreachable (the script falls back to "basic validation"), that is normal — the basic check is sufficient.

**Always run this before committing.** The build-and-deploy workflow runs the same script and aborts on failure, leaving the catalog stale.

### 7. Commit on the feature branch

```bash
git add datasets/<DATASET_NAME>/
git commit -m "Add <DATASET_NAME>"
```

Do NOT commit `uv.lock` — it's a side-effect of running `uv` in the registry clone and gets created if you've previously run the generator with project deps. The `--no-project` invocation avoids creating it; if it appears, leave it untracked.

### 8. Push the branch

This is where automation hits a wall. The Claude Code auto-mode classifier blocks pushes to repos outside the trusted source control of the current project, even with in-conversation user authorization. **You (the human) must run the push from your own shell**:

```bash
cd ../pad_dataset_registry
git push -u origin <BRANCH_NAME>
```

Or, in a Claude session, use the `!` shell prefix to run it without going through the classifier:

```
! cd ../pad_dataset_registry && git push -u origin <BRANCH_NAME>
```

### 9. Open the PR

`gh pr create` works inside Claude sessions once the branch is on the remote. Template:

```bash
gh pr create \
  --title "Add <DATASET_NAME>" \
  --base main \
  --head <BRANCH_NAME> \
  --body "$(cat <<'EOF'
## Summary
- <N> images / <M> unique sample IDs, PAD#-grouped split.
- <One paragraph on what's distinctive.>

## Standard schema + extensions
- 8-column schema: id, sample_id, sample_name, quantity, camera_type_1, url, hashlib_md5, image_name.
- Extended columns: <list any extras>.

## Independent verification
- md5 hashes of shared images match the prior <PRIOR_ENTRY> entry byte-for-byte.

## Local validation
- docs/_scripts/validate_croissant.py reports basic validation passed.

## Citation
> <Authors>. *<Dataset Title>, <Version>.* <Lab, Affiliation>. <Source repo URL>

## Test plan
- [ ] Registry CI rebuilds Jekyll catalog without errors.
- [ ] Catalog page renders the README + figs.
EOF
)"
```

### 10. Merge and verify

Have the right person review and merge. After merge:

- Within ~30 s, `Build and Process Dataset Catalog` triggers on `main`.
- Within ~1–2 min, it finishes and the `pages-build-deployment` job queues up the `gh-pages` push.
- Within another ~1 min, the catalog page is live at `https://padproject.info/pad_dataset_registry/datasets/<DATASET_NAME>/`.

If anything fails, check the Actions tab of the registry repo.

## Common gotchas (lessons learned the hard way)

| Gotcha | Symptom | Fix |
|---|---|---|
| Cross-org push blocked | "Permission for this action was denied by the Claude Code auto mode classifier" | Run the push yourself (`!` prefix or your shell). The classifier blocks cross-org pushes regardless of in-conversation authorization. |
| `nvidia-cublas-cu11` install error from `uv run` | Croissant generator fails before its first prompt | Add `--no-project` to the `uv run` invocation. |
| Generator hangs on first prompt | "EOFError: EOF when reading a line" | Pipe answers via `printf '...\n...\n...\n' \| python ...`. The generator asks exactly three questions in order: description, version, creator. |
| Dataset name and directory name diverge | Validation passes locally; published page shows wrong name | The croissant embeds the dataset name in `@id` and `contentUrl` paths. Re-run `create_croissant.py` after any rename and commit the regenerated file. |
| `figs/` not rendered on the catalog page | Images show as broken links on padproject.info | Verify the README references use `./figs/<name>.png` (relative). `generate_catalog.py` copies the `figs/` directory into the rendered output (lines 174–180); references must be relative. |
| Stale `gh-pages` after merge | Catalog page doesn't update | Check the `Build and Process Dataset Catalog` run in the Actions tab. The workflow only runs on push to `main` or `dataset_server` and only when files under `datasets/**`, `docs/**`, or `**.md` change. A failed validate aborts the deploy. |
| Schema mismatch warning | Local validator passes but the published Jekyll page renders some column wrong | The croissant's `datasetSchema.columns` describes the standard 8 columns; extended columns are silently ignored by strict consumers. Don't try to extend the croissant schema for the extensions unless you also want consumers to validate against them. |
| Author name disagreement | The croissant says one thing, the README says another | Update both at once. The catalog page rendering pulls from both; an inconsistency between them shows up on the published page. |
